# YOLO Inference Stack — TensorRT Implementation

## Overview

The YOLO object detection implementation now uses **TensorRT** for inference on both x86_64 and ARM64 (Jetson) architectures. This replaces the previous ONNX Runtime implementation which had compatibility issues with Jetson Nano's JetPack 4.6.

## Architecture

### Inference Backend: TensorRT

- **Desktop x86_64**: Uses TensorRT installed via NVIDIA packages (`libnvinfer`, `libnvonnxparser`)
- **Jetson ARM64**: Uses TensorRT pre-installed with JetPack (4.6 for Nano, 6.x for newer devices)
- **Engine Generation**: Runtime lazy generation with caching for fast subsequent startups
- **Precision**: FP16 enabled when supported by hardware

### Model Artifacts

| Artifact | Purpose | Location |
|----------|---------|----------|
| `yolov10n.onnx` | Authoritative model export | `data/models/` |
| `yolov10n.engine` | Pre-built engine (x86_64) | Generated at runtime |
| `yolov10n.jp46.engine` | JetPack 4.6 engine (Nano) | Generated at runtime |
| `yolov10n.jp6.engine` | JetPack 6.x engine (Orin/NX) | Generated at runtime |

## Build System

### Bazel Configuration

- `MODULE.bazel`: Uses `new_local_repository` for TensorRT system libraries
- `bazel/third_party/tensorrt.BUILD`: Defines `nvinfer` and `nvonnxparser` libraries
- `src/cpp_accelerator/infrastructure/cuda/BUILD`: `yolo_detector` depends on TensorRT

### Docker Build

1. **yolo-model-gen**: Exports YOLO model to ONNX format
2. **trt-engine-gen**: Generates TensorRT engine using `trtexec`
3. **cpp-built**: Builds C++ accelerator with TensorRT-linked libraries

## Implementation Details

### YOLODetector Class

The `YOLODetector::Impl` class now uses:

- `nvinfer1::IRuntime`: For deserializing pre-built engines
- `nvinfer1::ICudaEngine`: The optimized inference engine
- `nvinfer1::IExecutionContext`: For executing inference
- `nvonnxparser::IParser`: For ONNX→engine conversion (runtime generation)

### Engine Loading Strategy

1. Check for cached engine file (architecture/JetPack-specific)
2. If found, deserialize and create execution context
3. If not found, parse ONNX and build new engine
4. Serialize engine to disk for future runs

### Inference Flow

```
Input Image → cuda_letterbox_resize (GPU) → host memory
→ TensorRT IExecutionContext::enqueueV3 → output tensor
→ Postprocess/NMS → Detection results
```

## Development Workflow

### Building Locally (x86_64 with TensorRT)

```bash
# Ensure TensorRT is installed
# Ubuntu: sudo apt install libnvinfer-bin libnvinfer-dev libnvonnxparser-dev

bazel build //src/cpp_accelerator/infrastructure/cuda:yolo_detector
bazel build //src/cpp_accelerator/ports/shared_lib:libcuda_processor.so
```

### Cross-Compilation for Jetson

```bash
# Build on x86_64 host for ARM64 target
bazel build --platforms=//bazel/platforms:jetson \
    //src/cpp_accelerator/ports/shared_lib:libcuda_processor.so
```

### Manual Engine Conversion

```bash
# Convert ONNX to engine manually
./scripts/models/convert_onnx_to_engine.sh data/models/yolov10n.onnx data/models/
```

## Deployment

### Desktop x86_64

1. Install TensorRT via NVIDIA repository
2. Copy `libcuda_processor.so` and `.onnx` model to target
3. First run generates `.engine` file automatically
4. Subsequent runs load cached engine

### Jetson Nano (JetPack 4.6)

1. Transfer ARM64 build to Jetson
2. Ensure TensorRT is available (pre-installed with JetPack)
3. First run generates `.jp46.engine` file automatically
4. FP16 acceleration enabled by default

### Jetson Orin/NX (JetPack 6.x)

1. Same deployment as Nano
2. Engine tagged as `.jp6.engine`
3. Better FP16 performance than Nano

## Migration Notes (from ONNX Runtime)

### Removed Dependencies

- `@onnxruntime_gpu` (x86_64 GPU prebuild)
- `@onnxruntime_linux_aarch64` (ARM64 CPU prebuild)

### Added Dependencies

- `@tensorrt//:nvinfer` (TensorRT runtime)
- `@tensorrt//:nvonnxparser` (ONNX parser)

### Compatibility

- **JetPack 4.6**: Now fully supported via TensorRT (no cuDNN 8 requirement)
- **x86_64 GPU**: Performance typically better than ORT CUDA EP
- **ARM64 CI**: Works with system TensorRT on self-hosted runners

## Troubleshooting

### Engine Not Found

If you see "Failed to load cached engine", the detector will automatically build from ONNX. This is slower on first run but expected behavior.

### Architecture Mismatch

Engine files are architecture-specific. If you see deserialization errors:
- Delete the existing `.engine` file
- Let the detector rebuild it for the current architecture

### TensorRT Not Found

**Desktop**: Install via NVIDIA packages
```bash
wget https://developer.download.nvidia.com/compute/cuda/repos/ubuntu2404/x86_64/cuda-keyring_1.1-1_all.deb
sudo dpkg -i cuda-keyring_1.1-1_all.deb
sudo apt-get update
sudo apt-get install libnvinfer-bin libnvinfer-dev libnvonnxparser-dev
```

**Jetson**: TensorRT is pre-installed with JetPack. Verify with:
```bash
dpkg -l | grep nvinfer
```

## Performance Notes

- **First run**: Slow (engine generation from ONNX, ~10-30 seconds)
- **Subsequent runs**: Fast (engine deserialization, ~100ms)
- **FP16**: Enabled when supported (2x faster than FP32 with minimal accuracy loss)
- **Memory**: ~1GB workspace configured for engine building
