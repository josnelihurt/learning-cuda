<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→WebSocket | Untrusted WebSocket messages from backend signaling server |
| client→WebRTC | Untrusted ICE candidates and SDP from backend |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | T (Tampering) | WebSocket signaling | accept | Signaling server is trusted (same-origin), no user input in signaling path |
| T-04-02 | I (Information disclosure) | ICE candidates | accept | ICE candidates contain public IP addresses (expected for WebRTC), no PII |
| T-04-03 | D (Denial of service) | WebSocket connection | mitigate | Implement connection timeout, close connection on error, log failures |
| T-04-04 | E (Elevation of privilege) | SDP offer/answer | accept | SDP is media configuration, no authorization/privilege escalation |
| T-04-05 | R (Repudiation) | Session lifecycle | accept | Session IDs are ephemeral, no audit requirements for video streaming |

**Rationale:** WebRTC signaling operates in trusted same-origin environment. Main risk is DoS from stale connections, mitigated by proper cleanup (closeSession) and connection state management.
</threat_model>
